## Part 5 — Developer Content: Why Real-Time Collab for AI Chats Matters

### Blog Post / Dev.to Article

**Title:** Why Real-Time Collaboration Changes Everything for AI Conversation UIs

**Slug:** `why-realtime-collab-ai-conversation-ui`

**Target:** Dev.to, Hashnode, træk blog — developer audience building AI products

**Estimated read time:** 6 min

---

**Introduction (150 words)**

Most AI interfaces are built for one person. You type, the model responds, you think, you type again. The whole interaction is a private dialogue.

But the most valuable AI work — research, design exploration, product ideation — is rarely done alone. Teams huddle around a screen. Async collaborators pick up where others left off. A designer annotates an LLM output. A PM branches from a conversation to explore an alternative framing.

And yet, almost every AI conversation UI is fundamentally single-player.

@traek/collab is our answer to this. But building real-time collaboration for a spatial, tree-structured conversation is meaningfully different from building a shared document editor. This post explains the specific challenges and how we solved them.

---

**Section 1: The unique constraints of a branching conversation (400 words)**

A linear chat app and a branching canvas app have very different collaboration semantics.

In a linear chat: there is one thread. All collaborators are looking at the same sequence of messages. Conflict resolution is simple — either you show both messages, or you prevent concurrent edits.

In a spatial canvas: conversation is a tree. Two collaborators might be exploring entirely different branches simultaneously. Or they might both try to add a reply to the same node at the same time. Or one collaborator deletes a branch the other is actively reading.

These are not hypothetical — they are common workflows:

- **Research pair sessions:** Two researchers split the exploration tree, one going deep on one branch while the other explores an alternative
- **Design reviews:** A designer looks at one output while a PM branches from an earlier node to try a different prompt
- **Async handoffs:** One collaborator ends a session mid-tree; another picks it up hours later

A naive implementation (last-write-wins with optimistic locking) breaks all three. You need a data structure that handles concurrent writes gracefully, supports partial synchronization, and preserves the tree structure across disconnects and reconnects.

That's why we chose Yjs.

---

**Section 2: How CRDTs map to a conversation tree (300 words)**

A CRDT (Conflict-free Replicated Data Type) is a data structure designed for distributed systems where writes can happen concurrently without coordination. The core guarantee: two replicas that apply the same set of operations always converge to the same state, regardless of the order those operations were applied.

For a spatial conversation canvas, this is the right semantic:

- A node addition on one client and a node addition on another client should both survive a sync — you want both branches
- A node deletion followed by a position update (on different clients) should resolve gracefully — typically, deletion wins
- Node content updates during streaming should merge at the character level — you don't want a chunk collision to corrupt the output

We use `Y.Map` for the node registry. Each node's properties (content, status, position, parentId) are stored as entries. Yjs handles concurrent updates to any property with last-write-wins semantics at the property level, not the node level.

This means two collaborators can both update different properties of the same node (one updates position, another updates content) without either update being lost.

---

**Section 3: Presence and cursors — the soft real-time layer (250 words)**

Node sync is the hard real-time layer — it must be correct and conflict-free. Cursors and presence are the soft real-time layer — they're ephemeral, they don't need to persist, and a brief inconsistency is acceptable.

Yjs provides an Awareness protocol for this. Each client maintains a local state (current position, color, name) and broadcasts it to peers. Awareness states are not persisted — they live only for the duration of the session.

@traek/collab surfaces this as:

- **CollabPresenceBubbles** — avatar chips showing active peers, with overflow handling
- **CollabCursorsOverlay** — remote cursors rendered on the canvas, color-coded per user, with a fade-out after 3s of inactivity
- **CollabStatusIndicator** — connection state (connected / connecting / offline) with auto-reconnect

The cursor positions are in canvas coordinates, so they correctly reflect where each peer is looking regardless of their zoom level or viewport offset.

---

**Section 4: The useCollab hook (200 words)**

The entire collab layer is exposed as a single Svelte 5 hook:

```typescript
const collab = useCollab(engine, {
  serverUrl: 'wss://collab.yourserver.com',
  roomId: 'session-id',
  user: {
    id: currentUser.id,
    name: currentUser.name,
    color: '#6366f1'
  }
})
```

`collab.status` — reactive connection state (`'connected' | 'connecting' | 'offline'`)
`collab.peers` — reactive array of active peers with their positions

The hook must be called during component initialization (Svelte 5 `$effect` constraint). It handles WebSocket lifecycle, reconnection with exponential backoff, and cleanup on component destroy.

No manual WebSocket management. No manual Yjs document setup. One hook, reactive state.

**Conclusion (100 words)**

Real-time collaboration in a branching AI canvas is a harder problem than it first appears. The tree structure, streaming content, and spatial positions each add constraints that a simple shared-document model doesn't handle well.

CRDTs give you the right foundation — concurrent writes are safe by design. Yjs provides a battle-tested implementation. The @traek/collab module wraps this into a developer experience that fits the træk API: one hook, reactive state, composable components.

If you're building an AI product where teams work together — on research, design, or exploration — you probably need this. Try it free for 14 days.
